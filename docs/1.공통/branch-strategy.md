# 브랜치 전략

> 효율적인 브랜치 관리를 위해 모든 팀원이 반드시 준수해야 합니다.

## 🔵 main 브랜치

- **항상 배포 가능한 상태**를 유지합니다.
- 코드 리뷰 없이 main에 직접 push하는 것은 **금지**합니다.
- develop 브랜치를 통해 충분한 테스트를 거친 뒤 merge합니다.
- main에 merge되면 CI/CD를 통해 운영 환경으로 자동 배포됩니다.

---

## 🟢 develop 브랜치

- 개발 단계에서 기능을 통합하는 브랜치입니다.
- 각 feature 브랜치의 PR은 **반드시 develop으로** 요청합니다.
- 전체 기능이 모이고 테스트가 끝나면 main으로 merge합니다.
- 스테이징 서버가 있다면 develop 기준으로 배포됩니다.

---

## 🟡 feature/\* 브랜치

- 기능 단위로 생성하는 작업 브랜치입니다.
- 브랜치 이름은 다음 규칙을 따릅니다:
  - 예시:
    - `feature/login-page`
    - `feature/user-auth`
    - `feature/blog-editor`
- 기능 개발이 완료되면 develop 브랜치로 PR을 생성합니다.
- PR 시에는 변경 사항·테스트 결과·연관 이슈 번호를 반드시 작성합니다.

---

## 🛠 브랜치 생성 규칙

- 브랜치를 생성할 때는 **최신 develop을 기반으로** 생성합니다.
- 브랜치를 길게 가져가지 않습니다(권장: 1~3일 내 작업 완료).
- 충돌이 예상되는 경우 미리 팀원들과 공유하고 조율합니다.

---

## 🔁 merge 규칙

- PR은 최소 **1명 이상의 리뷰어 승인** 후 merge합니다.
- 충돌이 발생하면 **브랜치 작성자가** 해결합니다.
- merge 전에 `npm run build` 또는 백엔드 빌드를 반드시 확인합니다.

---

## 🚨 금지 규칙

- main, develop에 직접 commit / push 금지.
- 브랜치 이름을 임의로 변경하거나 강제 push하는 행위 금지.
- 테스트 없이 PR merge 금지.

---

## 📌 기타 권장 사항

- 큰 기능은 `feature/xxx` 안에 더 작은 단위로 쪼개서 작업해도 좋습니다.
  - 예시:
    - `feature/auth-login-ui`
    - `feature/auth-login-api`
- PR에는 스크린샷(프론트), API 응답 예시(백엔드)를 첨부하는 것을 권장합니다.
